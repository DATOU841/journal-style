## 5. 如何用 `source-role-register.json`（来源角色结构）

目标：识别并利用来源角色，但 **角色标签只作辅助证据，不得冒充全文风格结论**。

安全读取字段（白名单）：`schema`、`generated_at`，以及来源列表中每条的 `source_id`/`stable_id`、`title`、`year`、`journal`、`authors`、`source_need`、`gap_id`、`kb_dest`、`consumption_role`、`bucket_tags`/`role_tags`、`priority`。不读取任何正文/摘录字段。

角色 → journal-style 分析维度映射（建议作为 `coverage_tags` / `source_role_tags`）：

- 目标期刊生态样本：标 `target_journal_ecosystem`，进入期刊画像主样本候选与覆盖度统计。
- 同刊/同栏目/同主题样本：标 `same_journal` / `same_column` / `same_topic`，服务栏目结构与选题趋势的样本充分性判断。
- 理论锚点：标 `theory_anchor`，只服务「理论来源生态」描述，不计入目标期刊全文体例样本。
- 比较材料：标 `comparison_material`，服务横向比较支撑显化与多期刊对比，不混入目标刊核心样本。
- 一手材料：标 `primary_source`，服务材料偏好分析的「材料类型」证据，不等于方法证据。
- 方法/材料证据：标 `method_evidence` / `material_evidence`，服务材料与方法偏好分析。
- 边界材料或反例：标 `boundary_or_counterexample`，用于标注覆盖边界与反例，防止过度泛化。

下游用法与约束：

- 核心库筛选（§10 `select_core_library.py`）：角色标签只做 **小权重增益与解释**，不改变 25%-40% 核心库比例 gate，不替代六因子打分；理论锚点/比较材料/一手材料只加标签、默认不入「目标刊核心全文样本」。
- 期刊生态画像：聚成 `source_role_ecology_summary`（metadata 层），区分目标刊样本/比较材料/理论来源/一手材料/边界材料。
- 补采请求：`gap_id`/`source_need` 汇入给检索入库的补采建议（§9、§10 `handoff-to-jiansuo-ruku.md`）。

硬禁止：不把角色标签直接推导为全文体例/论证风格结论；不把 `sources/*.md` 正文片段当全文证据；角色缺失时所有依赖它的输出必须能优雅退回旧逻辑。

## 6. 如何用 Zotero 全量底表分析参考文献生态

必须区分两个范围（不得混同）：

- `02-kb-builder/zotero-reference-full-bibliography.md` = 任务全量候选/保存范围（含无 PDF、未进 025 队列、未正式入库条目），由 step2 元数据派生。
- `zotero-reference-master.md` = 正式入库/正式消费范围（属 writer 矩阵；journal-style 现状未消费，见 §3.2）。journal-style 侧实现时：master 若可得则直接用；不可得则以 §3.2 窄集交接面（`zotero-pdf-rag-handoff.json` 的入库计数 / `mu-fulltext-core-pack.json` 的 article 集）作「正式入库范围」代理，并在输出标注 `master_source=proxy`。

分析口径（只做公开元数据级解析，**不读 Zotero DB**）：

- 轻量 Markdown 条目解析，提取题名、年份、来源、文献类型等公开元数据；不依赖 citation key 作唯一真实凭据。
- 输出 `03-analysis/metadata-layer/bibliography-scope-coverage.json`，比较 full vs master 的：条目数量差、可解析题名差、年份分布差、来源/文献类型分布差、期刊内互引线索（仅在元数据可判时）、正式入库覆盖率、未入正式池候选的安全题名清单。

帮助 journal-style 判断：样本覆盖是否偏窄、参考文献生态广度、补采需求（哪些候选值得推动正式入库）。所有覆盖差与缺口同时写入文衡状态与（若有）gap 写回（§9）。底表/对比结论只属 metadata 层证据，不得单独支撑全文体例/参考文献生态高置信结论（仍受 §3.1 证据门槛约束）。

## 7. `fulltext-index.json` / `manifest.json` 边界 + 保密边界扩展

### 7.1 默认只读字段（指针级，不开正文）

## 8. 如何用 `rag-query-seed-pack.json`（转查询种子计划，不执行 RAG）

目标：把上游 query seeds 转为 journal-style 的「RAG 查询种子计划」，**只生成计划，不执行 RAG、不连接向量库、不读取 chunk**。

安全读取：查询主题/意图、维度标签、seed terms/query strings、`source_id` 指针、`gap_id` 指针。

按 journal-style 关心的维度组织种子计划（覆盖任务要求的 7 类）：题名风格；摘要/关键词结构；材料与方法偏好；章节层级与格式体例；论证节奏；注释/参考文献生态；投稿运营与格式证据。

产出：

## 9. 如何用 `gap-ledger.json`（降低投稿匹配过度确定）

目标：用上游缺口登记，**降低投稿匹配判断的过度确定**，并驱动降级与补采，而不是把缺口伪装成高置信结论。

读取并归类缺口：缺全文、缺正式入库、缺目标期刊样本、缺理论锚点、缺参考文献解析、缺格式证据等。

降级联动（与 §3.1 证据门槛、`evidence-rules.md` 降级表对齐）：

- 缺全文 / MinerU/mu 完整全文 <10 篇：禁止进入下游体例约束锁；全文体例只写待补。
- 缺正式入库 / RAG 可用率不足：论证风格、参考文献生态降为「待全文证据补足」，不给高置信。
- 缺目标期刊样本 / 样本 <20 篇：不得写全刊稳定风格。
- 缺参考文献解析 <30 篇：参考文献生态不给高置信。
- 缺格式证据：格式体例只保留投稿须知/题录层判断。
- 投稿匹配评分：在 `submission-fit` 输出显式带 `gap_adjusted=true` 与受影响维度，存在硬缺口时不得输出高适配强结论。

写回与防伪装：

- 把合并后的缺口（sidecar `gap-ledger` ∪ journal-style 自有 gate 缺口）输出补采优先级，并经 `handoff-to-jiansuo-ruku.md` 出站请求字段（`sidecar_gap_ids`/`role_register_gaps`/`fulltext_index_gap_summary`/`bibliography_scope_gap_summary`）回写给检索入库；仍只生成请求，不执行检索入库。
- 缺口状态必须同步进文衡状态/H08；不得用缺口存在的任务伪装成「投稿匹配已完成 / 高置信」。

## 11. fail-closed vs warning/degraded 矩阵

明确「哪些 gate 必须 fail-closed」「哪些 sidecar 缺失只能 warning/degraded」：

必须 fail-closed（NO_GO / 阻断），均为既有红线，sidecar 适配不得削弱：

- full 模式既有硬 gate 不变：`mu-fulltext-pack`、`per-article-profile-complete`、`provenance-required`、`scoring-replay-calibrated`、`no-metadata-only-completion`、run-mode path gate、core-library 25-40%。
- `release-integrity`（config/脚本 sha 漂移）、文衡原生绑定校验：保持 fail-closed。

仅 warning/degraded（不阻断主流程）：

- 任意 sidecar **缺失**：`jiansuo-sidecar-safety` 判 PASS（`missing_sidecar_is_pass=true`），Step0 记 `optional_gaps` 而非 hard/soft gap，主流程回退 §3.2 窄集面继续。
- sidecar 存在但 schema 不识别/字段缺失：schema-tolerant，忽略未知字段，记 warning，不 NO_GO。
- `downstream-consumption-manifest.json` 声明「可消费」但与真实文件/sha 不一致：记 warning + 以真实文件为准，不把其 ready 提升为 journal-style PASS。
- source role / fulltext-index / bibliography 部分可用：对应增强降级为「部分可用」，core library 等回退旧逻辑，不阻断。

红线一致性：sidecar 适配 **只能新增 nonblocking 增强 + 一个泄漏 fail-closed 安全 gate**；不得把任何 sidecar 缺失变成新 hard gate，也不得用 sidecar 存在绕过 full 模式既有硬 gate。
