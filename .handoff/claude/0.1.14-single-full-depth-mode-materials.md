# journal-style 0.1.14 单一全量深度模式材料整理

## 业务问题

`journal-style` 更新后，文衡 C03「刊程追踪」未显示最新期刊画像。已确认不是前后端完全未衔接，而是 skill 仍允许 `standard` / `metadata-only` 作为任务终点，导致期刊画像以非生产 profile 的形式被隔离，不能进入正式列表。

## 已核实事实

- 文衡后端存在 C03 写入和读取接口：`POST /api/c03/journal-profiles/from-task/:taskId` 与 `GET /api/c03/journal-profiles`。
- 生产运行文件 `/opt/wenheng-control/repo/data/runtime/journal-profiles-runtime.json` 中已有 `江汉论坛` 记录，但该记录 `source_run_id=null` 且 `production_profile=false`，标签包含 `metadata-only` 与 `standard-mode`。
- 该记录摘要明确写为 metadata 层画像，不提供全文体例和参考文献生态稳定结论，因此按 C03 正式过滤规则不应进入正式画像列表。
- 当前 skill 文档、状态机和入口仍保留 `light`、`standard`、`full` 三模式，其中 `standard` 是默认模式，且不要求 MinerU/mu 完整全文包。
- `江汉论坛` 任务后续已经补齐全文 sidecar：服务器路径 `/home/ubuntu/wenheng-runtime/jianghan-forum-journal-style-20260704/tmp/cnki/20260704-133757-query-d2130a/025-rag-import/fulltext/` 下 54 个 `full.md` 均为非空。
- 但 full-depth journal-style 链未继续生成并登记 `mu-fulltext-core-pack.json`、`per-article-style-profiles.json`、`journal-style-aggregation-bundle.json`、`journal-polish-consumption-pack.json`、`wenheng-center-status.json` 与 C03 server-controlled sync receipt。

## 根因判断

根因不是 C03 展示组件单点问题，而是 `journal-style` 交付契约本身仍有两个出口：metadata-only 可以结束，full-depth 也可以结束。正式文衡任务中，这会制造看似完成但不能进入正式 C03 的中间产物，后续下游也无法判断该任务是否真的交付成功。

## 修复目标

- 正式 `journal-style` 任务只有单一 full-depth 模式。
- `metadata-only` 只能作为阻塞或中间状态，不能作为正式完成、正式 C03 profile 或下游交付。
- 默认运行必须进入 full-depth 路径，缺 MinerU/mu 完整全文包时必须停在全文包验收步骤。
- `light` / `standard` 不再作为正式模式存在；历史或离线调试也不得生成 production evidence。
- C03 profile 只允许在 Step10 全链完成并带有 `source_run_id`、H08 evidence、source lock 后写入。
- `江汉论坛` 应从现有 54 个非空 `full.md` 继续完成 full-depth 链，不应重跑 CNKI/Zotero/PDF/RAG。

## 需重点检查的代码位置

- `SKILL.md` 的运行模式章节。
- `references/run-modes-protocol.md` 的三模式协议。
- `references/stage-gates-protocol.md` 与 `references/output-schema.md` 中对 `light` / `standard` 作为 metadata terminal 的说明。
- `config/workflow-states.json` 的 `run_modes` 与 `step08c_metadata_handoff`。
- `scripts/build_task_skeleton.py` 的 `--run-mode` 默认值和初始状态。
- `scripts/journal_style_runner.py` 的 `resolve_run_mode()`、`workflow_steps_for_mode()` 和 `--mode` 参数。
- `tests/run_state_machine_fixtures.py` 中 `metadata_mode_reaches_metadata_terminal` 等回归夹具。

## 验收标准

- 默认 skeleton 和 runner 均解析为 `full`。
- 旧 `light` / `standard` 参数不能进入正式路径；生产任务不会因旧字段回落到 metadata terminal。
- 无 MinerU/mu 完整全文包时，runner 停在 `step07b_mu_fulltext_pack`。
- 有完整全文包、逐篇画像、聚合包、校准评分和 handoff 产物时，runner 能到 `completed`。
- 状态机配置中不存在正式 `metadata_terminal` 路径。
- 文档不再宣称 `light` / `standard` 可作为正式完成模式。
- 回归测试覆盖“旧 standard 状态被强制转 full 并阻塞在全文包验收”。
